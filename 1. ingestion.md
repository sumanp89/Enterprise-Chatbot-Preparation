# Phase 1: Source Processing — Data Extraction, Routing & Ingestion Pipelines

At a scale of millions of daily users, handling complex visual structures (diagrams, 3D assets) and unstructured data requires moving past naive text scripts into structured document engineering.

This document covers the complete source processing layer: image handling, HTML/PDF parsing, tabular data extraction, file routing, and the three ingestion lanes (Static, 6-Hour Scheduled, Near-Real-Time).

---

## Part 1: Data Extraction & Content Handling

### 1. Image Processing: Diagrams vs. 3D Assets

Images within HTML and PDFs must be handled based on their **semantic intent**. Mismanaging this wastes vector storage and breaks user trust.

#### A. Sequence Diagrams, Code Frameworks & Flowcharts

Converting sequence diagrams into raw ASCII text is highly unreliable for RAG. Standard tokenizers break ASCII graphics into broken patterns, destroying context.

- **Solution:** Convert the image into standard **Mermaid.js** syntax or structured **Markdown** tables using a Vision-Language Model (VLM).
- **Why it works:** LLMs natively understand Mermaid markdown strings (e.g., `graph TD; A-->B;`). The underlying text contains semantic words like "requests," "sends," and "verifies," which the vector database can match against.

#### B. High-Fidelity 3D Images, Product Assets & Schematics

Multi-angle 3D schematics or high-definition product views cannot be represented purely as text. Converting solely to a description loses the rich visual context.

**Solution — Image Multi-Route Strategy:**

1. **Extract and Host:** The parsing worker cuts out the image, hashes it to avoid duplicates, uploads to a secured CDN (AWS S3 / GCP GCS), and attaches a hard-linked URL.
2. **Generate a Context Wrap:** Pass the isolated image through a fast, lightweight VLM to generate a detailed semantic caption.
3. **Embed the Wrap:** Store only the generated text caption inside the Vector Database chunk, but bundle the **CDN image URL in the metadata**.
4. **Payload Return:** When a user queries something relevant, the text description triggers the vector match. The chatbot presents the textual answer alongside the hosted image URL, allowing the frontend to render the complete visual component.

### 2. HTML Processing: The BeautifulSoup Pipeline

BeautifulSoup constructs a tree-like object model (`Tag` objects, parents, children). It does **not** handle chunking — it serves as a structural filter *before* handing data to the chunking logic.

Passing raw HTML directly to a text splitter will slice through markup, producing broken fragments like `<div>Clas` and `s="content">`.

#### Pipeline Flow

```
[Raw HTML String] ──► BeautifulSoup (DOM Filtering) ──► Node Traversal (Semantic Parsing) ──► Clean Markdown Chunks
```

1. **Sanitization:** Strip all non-content noise (`<script>`, `<style>`, `<nav>`, `<footer>`).
2. **Structural Extraction:** Instead of flattening with `.get_text()`, write an AST walker that converts semantic HTML blocks into clean Markdown:
   - `<h1>`, `<h2>` tags become `#`, `##`.
   - `<table>` sections are isolated and converted into Markdown or HTML string grids.
3. **Hierarchy-Aware Chunking:** Use the clean Markdown with a structural splitter that slices at the boundaries of headers or main structural blocks, guaranteeing structural context stays intact.

### 3. Open-Source Document Layout Models

For local deployment to control data privacy and infrastructure costs, specialized open-source VLMs and pipelines process PDF pages as a complete visual canvas and output structured markdown.

| Model / Framework | Core Architecture Base | Best Use Case |
| --- | --- | --- |
| **OlmOCR-2** / **DeepSeek-OCR** | Fine-tuned Vision LLMs | Scanned pages, forms, and handwritten text. Single visual forward-pass. |
| **Qwen2.5-VL / Qwen3-VL** (4B to 72B) | Native Vision-Language | State-of-the-art layout comprehension, table extraction, chart-to-markdown. |
| **Docling** (by IBM Research) | Layout Engine + Local Models | Lightweight, purpose-built for fast PDF-to-JSON structural parsing on custom container infrastructure. |
| **MinerU / Marker** | Multimodal Layout Pipelines | Dense scientific/technical whitepapers with multi-column blocks and embedded math. |

#### Local Infrastructure Deployment

Using frameworks like **vLLM** on self-hosted GPU node instances, these layout models are exposed as high-throughput internal processing endpoints. When a document arrives via the processing queue, the worker converts PDF pages to high-resolution images, runs them through the local vision model endpoint for structured Markdown output, and pipes it into the semantic chunking layer.

### 4. Handling Code Syntax in PDFs and HTML

Code blocks inside documents are notorious for losing formatting during naive extraction. Indentations disappear and special characters (`<`, `>`, `{`, `}`) get garbled.

#### In HTML:

Code is almost always wrapped inside `<pre><code>...</code></pre>` tags.

- **Pipeline Action:** The BeautifulSoup parser explicitly looks for these tags. When found, it extracts the inner text, detects the language class attribute (e.g., `class="language-python"`), and wraps it cleanly in Markdown backticks with the language identifier:

````markdown
```python
def calculate_yield(principal, rate):
    return principal * rate
```
````

#### In PDFs:

PDFs don't have tags unless explicitly generated as "Tagged PDFs" (which is rare). Standard text extractors like `PyPDF` strip out whitespaces, making Python or Go code unreadable.

- **Pipeline Action:** Layout-aware engines or local VLMs (like **Docling** or **Qwen2.5-VL**) process the document *visually*, preserving whitespace indentations and wrapping code structures inside Markdown code fences automatically.

### 5. Handling Tabular Data

If a table is flattened into a single string (e.g., `Year 2025 Revenue $10M Year 2026 Revenue $15M`), an LLM will struggle to map relationships correctly.

```
       Naive Flattening:    "Year 2025 Revenue $10M Year 2026 Revenue $15M..." (Fuzzy)
       Markdown/HTML Table: "| Year | Revenue |" -> Explicit semantic boundaries for the LLM.
```

#### Ingestion Blueprint for Tables:

1. **Extract as Structured Strings:** Tables must be extracted as **Markdown Tables** or **HTML `<table>` strings**. Both structures inherently tell the LLM exactly which data point belongs to which column and row header.
2. **Context Enrichment Pass:** A table isolated by itself can lose meaning if it relies on surrounding text (e.g., a table listing numbers without mentioning it applies to the "Singapore Branch"). The parser must prepend **Parent Section Headers** to the table chunk.
3. **Multi-Row Strategy for Huge Tables:** If a table spans 3 pages, do not chunk it blindly in the middle. Slice the table horizontally by rows, and ensure the **Column Headers are injected into the top of every sub-chunk**.

### 6. Automated File Identification & Routing

When a file hits the enterprise landing zone queue, file extensions cannot be trusted (e.g., renaming a `.png` to `.pdf` happens). A **two-tier classification funnel** determines processing.

```
                  ┌──► Mime-Type: text/html ──► HTML Engine (BeautifulSoup)
                  │
[Inbound File] ───┼──► Mime-Type: application/pdf
                  │           │
                  │           ▼
                  │    PDF Complexity Inspector (Sample Metrics)
                  │           │
                  │           ├──► Native/Text-Heavy? ──► High-Speed Text Parser
                  │           │
                  │           └──► Layout/Image-Heavy? ──► Vision Layout Engine (Docling/Qwen)
```

#### Tier 1: True Type Determination (Magic Bytes Check)

The intake worker inspects the file's **hexadecimal signatures (magic bytes)** rather than the file extension using a library like `python-magic` (which interfaces with the Unix `file` command).

- Signature = `%PDF-` --> **PDF Pipeline**.
- Signature matches text/html structures --> **HTML Pipeline**.

#### Tier 2: PDF Complexity Inspector

Once identified as a PDF, a rapid lightweight inspection script reads the first 3-5 pages to determine if it's text-heavy (cheap to process) or image/table-heavy (expensive vision models needed):

```python
import pypdf

def inspect_pdf_complexity(file_path):
    reader = pypdf.PdfReader(file_path)
    total_pages = len(reader.pages)
    
    sample_pages = reader.pages[:5]
    
    total_text_length = 0
    total_images_count = 0
    
    for page in sample_pages:
        total_text_length += len(page.extract_text() or "")
        total_images_count += len(page.images)
        
    avg_text_per_page = total_text_length / min(5, total_pages)
    avg_images_per_page = total_images_count / min(5, total_pages)
    
    # Dynamic Routing Logic
    if avg_text_per_page < 200 or avg_images_per_page > 1.5:
        return "IMAGE_HEAVY_OR_SCANNED"
    else:
        return "TEXT_HEAVY_NATIVE"
```

## Part 2: The 6-Hour Scheduled Pipeline

Every 6 hours, the system must efficiently crawl, inspect, filter, chunk, and update thousands of files without reprocessing unchanged data (wasting money) or missing updates.

### Architecture Blueprint

```
[EventBridge Cron] ──► AWS Step Functions (Orchestrator)
                                │
                        [Distributed Map] 
                                │
        ┌───────────────────────┴───────────────────────┐
        ▼                                               ▼
[Delta Engine (MD5 Check)]                      [Processing Worker]
• Read Source S3 vs DB State                   • Extract (BeautifulSoup/VLM)
• Identify: CREATE / UPDATE / DELETE            • Vector & Metadata Upsert
```

### 1. Orchestration via AWS Step Functions (Distributed Map)

At enterprise scale, passing an array of thousands of files through a regular Step Function will crash due to the **256 KB state payload execution limit**.

The **Step Functions Distributed Map state** in **Express Workflow mode** solves this:

- The Step Function doesn't read file data into memory. Instead, it uses an `ItemReader` targeted at an S3 prefix or a manifest file.
- **High Concurrency:** Distributed Map spawns child executions (up to 10,000 parallel) to process batches independently, completing the 6-hour sync in minutes rather than hours.

### 2. Delta Engine: MD5 Hashing & State Comparison

Expensive AI extraction or chunking should never run on an unchanged 50MB PDF. A custom **State Registry** backed by **Amazon DynamoDB** implements efficient change data capture (CDC).

#### Tracking Schema (DynamoDB `knowledge_file_registry`)

- **Partition Key (`PK`):** `FILE#<Hash_or_Path>` (e.g., `FILE#wiki/compliance_sg_2026.pdf`)
- **Attributes:** `current_md5_checksum`, `last_processed_timestamp`, `associated_chunk_ids` (String Set).

#### Deterministic State Flow

When a file item is processed by a Distributed Map child task:

1. Read the source file's metadata from S3. S3 automatically calculates an `ETag` (often the MD5 hash for single-part uploads) or we compute the MD5 block hash.
2. Query the DynamoDB state registry.
3. **Branching Logic:**
   - **Scenario A (New File):** MD5 not in DynamoDB --> Route to **Full Processing** --> Action = `CREATE`.
   - **Scenario B (Unchanged File):** Inbound MD5 matches DynamoDB MD5 --> **Short-circuit instantly**. Zero processing or LLM cost.
   - **Scenario C (Modified File):** Inbound MD5 does *not* match DynamoDB MD5 --> Route to **Full Processing** --> Action = `UPDATE`.

### 3. Upsert & Purge Logic

An `UPDATE` action is risky in RAG. If an old document had 10 chunks and the updated version has 8, simply upserting by file ID will leave 2 orphaned chunks in the Vector DB. The chatbot will read outdated data.

**Solution: Delete-Before-Write Transaction Pattern:**

```
                       ┌──► 1. Read Old Chunk IDs from DynamoDB
                       │
[UPDATE/DELETE Event] ─┼──► 2. Purge Old Chunk IDs from Vector DB (Hard Delete)
                       │
                       └──► 3. Generate New Chunks ──► Write to Vector DB & Update DynamoDB
```

#### Step 1: Isolation & Purge

For `UPDATE` or `DELETE` actions, fetch the `associated_chunk_ids` from DynamoDB. Fire a batch-delete request to the Vector Database (Pinecone/Milvus) to strip those specific vector IDs from the index.

#### Step 2: Processing & Generation (CREATE and UPDATE)

The clean file goes through structural layout parsers (BeautifulSoup for HTML, Docling/Vision models for image PDFs) to produce new semantic chunks.

#### Step 3: Atomic State Registration

Push new vector chunks into the Vector Database. On success, perform an atomic write back to DynamoDB saving the **new MD5 hash** and overwriting the `associated_chunk_ids` with newly generated IDs.

---

### MD5 Hashing Cost for Large Files

**Cheap and fast.** Running MD5 or SHA-256 over a 100MB file takes roughly **50 to 100 ms** with negligible CPU.

**Production Trick: Never download the file just to hash it.** Since data lives in S3/GCS:

- When uploaded to S3, AWS automatically calculates an **`ETag`** (the MD5 hash for standard uploads; a composite MD5 of parts for multi-part uploads).
- The Step Function worker makes a metadata-only API call (`s3.head_object`). The storage provider returns the hash instantly without transferring a single byte.

### Why Delete-and-Insert Beats In-Place Update

In traditional databases, in-place `UPDATE` is standard. **In Vector Databases, Delete-then-Insert is universally faster, safer, and cleaner.**

**Why in-place updates fail:**

Suppose `v1` of a PDF yields 10 chunks, and `v2` yields 8 (text was removed). In-place `UPSERT` by mapping chunk positions (`chunk_0` through `chunk_7`) successfully overwrites the first 8 chunks. However, **`chunk_8` and `chunk_9` from the old version still exist** — they become orphaned. The chatbot will return stale data.

**Strategy:** Purging by exact ID sets (`VectorDB.delete(ids=[...])`) and pushing new arrays (`VectorDB.upsert(vectors=[...])`) takes milliseconds and guarantees no legacy fragments remain.

---

### Handling Deleted Files: Multi-File Snapshot Comparison

When an admin deletes 5 files from a directory of 30, a standard crawl returns only 25 items. The system won't "see" the 5 missing files unless an explicit sync pattern is implemented.

Instead of relying on webhooks or file renaming (which administrators forget), the Step Function performs a **differential loop** during the 6-hour cron:

```
Step 1: Fetch all current file keys from S3 Folder -> [A, B, C]
Step 2: Scan DynamoDB Registry for that folder       -> [A, B, C, D, E]
Step 3: Diff Array Math                              -> [D, E] are missing!
Step 4: Trigger Purge Worker for keys [D, E]
```

1. **Discovery Scan:** List all *active* file strings from the corporate directory or S3 bucket.
2. **Registry Query:** List all files currently registered in DynamoDB for that path.
3. **Logical Difference:**

$$\text{Deleted Files} = \text{DynamoDB Records} \setminus \text{Active Storage Files}$$

4. **Cleanup Loop:** Files found in DynamoDB but missing from physical storage are immediately routed to a deletion worker. The worker purges associated vectors and removes the tracking row from DynamoDB.

---

### DynamoDB as Single Source of Truth

Vector databases are fast search indexes — they are **not** relational data registries. They are terrible at managing state, complex aggregations, or tracking configurations. DynamoDB acts as the master operational brain.

#### Production DynamoDB Row State Example

```json
{
  "PK": "FILE#corporate/policies/abc.pdf",
  "file_metadata": {
    "file_name": "abc.pdf",
    "storage_path": "s3://enterprise-knowledge-base/corporate/policies/abc.pdf",
    "file_size_bytes": 104857600,
    "last_modified_at": "2026-06-20T14:22:18Z",
    "checksum_etag": "702242d3703818ddefe6bf7da2bed757"
  },
  "processing_stats": {
    "total_chunks_generated": 30,
    "parsing_engine_used": "DoclingLayoutVision_v2",
    "embedded_with_model": "text-embedding-3-large",
    "sync_latency_seconds": 12.4
  },
  "associated_chunk_ids": [
    "chnk_abc_001",
    "chnk_abc_002",
    "chnk_abc_003",
    "chnk_abc_030"

  "security_rbac_fallback": {
    "permitted_roles": ["hr_admin", "executive_staff"],

```

**Disaster Recovery Benefit:** If the Vector Database index gets corrupted, goes offline, or the provider is swapped (e.g., migrating from Pinecone to Milvus), there is no need to re-parse raw text/PDF archives. A recovery script reads DynamoDB state, loops through stored chunks, and streams embedded states back into the new index.

## Part 3: The Near-Real-Time (NRT) Lane

For data updates that cannot tolerate a 6-hour delay (e.g., live stock alerts, critical policy changes, urgent system status), standard crawling fails. At a scale of millions of users, NRT ingestion must switch from **polling** to a purely **reactive, event-driven streaming architecture**.

**Objective:** Sub-second to sub-minute latency from data change event to vector availability, without overwhelming downstream embedding models.

### Architecture Blueprint

```
                    [Change Data Capture (CDC) / Webhook]
                                      │
                                      ▼
                       Event Backbone (Kafka / Pub/Sub)
                                      │
                         Stream Consumer / Worker
                                      │
            ┌─────────────────────────┴─────────────────────────┐
            ▼                                                   ▼
   [In-Memory Deduplicator]                            [Atomic Vector Swapper]
   • Redis Window Lock                                 • Lock-free Vector Overwrites
   • Drops rapid multi-saves                           • Live metadata update
```

### 1. Real-Time Trigger Mechanics

No directory scanning for NRT data. The application layer or database pushes updates using one of two patterns:

- **Change Data Capture (CDC):** If data is backed by an operational database (PostgreSQL, MongoDB, CMS backend), a log-scanner like **Debezium** is deployed. The moment a row is updated, the transaction log emits an event with `Before` and `After` states.
- **Application Webhooks:** External platforms or internal microservices fire structured transaction events into the API Gateway.

### 2. Traffic Cushion: Event Streaming Backbone

When thousands of data points update concurrently, firing lambdas or calling embedding APIs directly triggers severe rate-limiting ($429$ Too Many Requests) or uncontrolled cost.

Events pass into an event broker: **Amazon Kinesis**, **GCP Pub/Sub**, or **Apache Kafka**.

- **Buffer Benefit:** The broker acts as a shock absorber. If a massive batch update occurs, events queue up safely. Processing workers consume them sequentially at an optimal throttle rate matching the maximum concurrency of parsing and embedding models.

### 3. Mitigating "Save Spam" (In-Memory Deduplication)

In real-world enterprise workspaces, authors updating live markdown files or scripts emitting HTML modifications often hit "Save" 5 times within 10 seconds. If the NRT pipeline responds to every click, it runs 5 parallel layout parsers and writes 5 conflicting vector payloads for the same data.

**Solution: In-Memory Sliding-Window Lock powered by Redis:**

```
Event Inbound (File A) ──► Check Redis Key "lock:FileA" ──► Found? ──► Drop Event (Duplicate)
                                  │
                             Not Found? ──► Set Key (TTL 5s) ──► Process Stream
```

1. Consumer pulls event for `doc_id_101`.
2. Executes an atomic `SETNX` (Set if Not Exists) in Redis for `lock:doc_id_101` with a TTL of **5 to 10 seconds**.
3. If the lock is acquired, the file proceeds to immediate processing.
4. If the lock fails (identical event is already processing or just processed), the incoming event is safely dropped as redundant. This drastically cuts resource expenditure.

### 4. Atomic Vector Swapping (Zero-Downtime Hot Updates)

For the 6-hour batch lane, a "Delete-Before-Write" pattern works. For the NRT lane serving millions of active users, a multi-step delete then insert creates a race condition — if a user queries in the millisecond window *after* a delete but *before* the new write completes, the answer is missing entirely.

**Solution: Atomic Vector Overwrite Strategy:**

- **ID Mapping Consistency:** Each sub-chunk within a near-real-time file maps to a deterministic string ID format (e.g., `nrt_<file_id>_chunk_<index>`).
- **Dynamic Update:** When the file is updated, the worker immediately updates the exact same structural IDs. Vector stores like Pinecone or Milvus execute an internal, atomic `UPSERT` on ID matches. The vector index switches the old array for the new array instantaneously, ensuring zero downtime or context gaps for live user interactions.

---

With the NRT lane established via event buffering, sliding-window deduplication, and atomic hot-swapping, all three data speeds (Static, Scheduled, and Near-Real-Time) are mapped for Phase 1.
