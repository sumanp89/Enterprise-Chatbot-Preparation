# Phase 3.1: Dynamic System Prompt Assembly

At a scale of **1M steady to 10M peak daily users**, a single monolithic system prompt for every query causes three major production failures:

1. **Cost and Latency Trap:** A 2,000-token system prompt processing 10M queries/day wastes **20 billion input tokens per day** on static instructions.
2. **Context Drift:** When an LLM is overloaded with rules for multiple roles (IT support *and* billing refund rules), its attention scores spread thin. The model confuses instructions or hallucinates cross-persona behaviors.
3. **Security Vulnerability:** Mixing internal employee guidelines and public-facing customer guardrails in the same instruction block increases the risk of a user extracting restricted internal instructions via prompt injection.

The solution is a decoupled, context-driven runtime assembly pipeline.

---

## Architecture

Instead of a monolithic prompt, the system prompt is assembled at runtime using small, distinct building blocks stored in an ultra-fast in-memory cache (**Redis**).

```
                      ┌──► Context Store (Redis) ──► Fetch Base Core Guidelines
                      │
User Request Token ───┼──► Role Identifier ───────► Fetch Role Persona Fragment (e.g., Finance Admin)
                      │
                      └──► Intent Classifier ─────► Fetch Domain Task Fragment (e.g., Billing Flow)
                                                    │
                                                    ▼
                                      [Dynamic Assembly Engine]
                                                    │
                                                    ▼
                                  Combined Target System Prompt Block
```

### The 3 Core Pillars

1. **Core Base Layer (Static & Global):** Non-negotiable enterprise rules — structural output formats (e.g., "Always reply in clear Markdown"), language constraints, and global safety rules.
2. **Persona Fragment Layer (RBAC-Driven):** Injected based on the user's validated identity token (JWT). Regular customers get customer service boundaries. Premium users get specialized service flows. Internal employees unlock technical internal-facing instructions.
3. **Intent/Domain Fragment Layer (Context-Driven):** Determined by a lightweight classification step that analyzes the user's incoming query before the generation LLM is called.

---

## Inbound Intent Router: Fast Classification

To determine which Domain Fragment to fetch without adding hundreds of milliseconds of latency, a **tiered routing architecture** is used instead of a heavy LLM for classification:

```
                       [Inbound User Query]
                                │
             Is it a clear match or simple pattern?
                                │
                 ┌──────────────┴──────────────┐
                 ▼                             ▼
         [Yes: Deterministic]          [No: Semantic]
          • Regex Token Map             • Fast Semantic Router 
          • Exact Path Lookup           • Lightweight Token Model (e.g., 1B-3B)
                 │                             │
                 └──────────────┬──────────────┘
                                ▼
                   Identified Domain Routing
```

- **Tier 1: Deterministic Routing:** For structural paths (e.g., user clicks a "Billing History" menu button), the frontend explicitly passes an intent flag parameter. The system short-circuits straight to the cache for the billing text fragment.
- **Tier 2: Algorithmic Semantic Routing:** For unguided text input, the query passes through a fast semantic router framework (like **Semantic Router** or a 1B-3B parameter classifier model). This maps the query to a category (e.g., "Technical_Support") within **< 15 ms**, bypassing the heavy primary generation model.

## Runtime Payload Assembly Example

An authenticated user with `internal_admin` role queries: *"Check policy guidelines for user account lockout remediation steps."*

### Step 1: Cache Fetch

The assembly layer fires parallel async fetches to Redis:

```python
base_guidelines = redis.get("prompt:core:base")
role_persona   = redis.get("prompt:persona:internal_admin")
domain_task    = redis.get("prompt:domain:compliance_policy")

# The retrieval pipeline from Phase 2 passes the verified chunks
retrieved_context_chunks = get_verified_rag_chunks(query, user_roles)
```

### Step 2: Structural Interpolation

The engine wraps blocks inside clear XML-tagged structural delimiters. LLMs are highly responsive to XML tags because they clearly demarcate where instructions end and where reference data begins:

```xml
<system_instructions>
{base_guidelines}

<identity_persona>
{role_persona}
</identity_persona>

<task_constraints>
{domain_task}
</task_constraints>
</system_instructions>

<verified_reference_context>
{retrieved_context_chunks}
</verified_reference_context>
```

### Why This Structure Protects Production

1. **Token Efficiency:** If the domain is compliance, the LLM receives zero billing tokens, zero shipping tokens, and zero technical support tokens. Prompt length stays thin, protecting throughput limits.
2. **Clear Boundaries:** Grouping RAG data inside `<verified_reference_context>` tags and telling the `<identity_persona>` layer to *only* answer using data inside those boundaries builds a robust programmatic defense against prompt injection tactics targeting data leaks.
