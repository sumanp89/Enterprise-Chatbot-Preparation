# Phase 2: High-Performance Advanced RAG Retrieval

With the data layer clean, chunked, cataloged in DynamoDB, and synced to the vector store, this phase covers the core retrieval engine: hybrid search, pre-retrieval query transformation, RBAC-aware filtering, and post-retrieval re-ranking.

At scale, a naive vector query like `db.search(query_embedding)` is a production anti-pattern.

---

## 1. Vulnerabilities of Single-Method Search

### A. Pure Dense Vector Search (Semantic)

- **Mechanism:** Converts text to high-dimensional floating-point arrays and performs ANN graph traversal (usually HNSW).
- **Failure Mode:** Struggles with **exact identifiers**. If a user types a specific error code `ERR_78492_A`, a configuration parameter `max_workers_count`, or a serial number, dense vector math smooths these terms into general semantic concepts. The exact match is completely missed, and the LLM receives the wrong context chunk.

### B. Pure Sparse Search (Lexical Keyword / BM25)

- **Mechanism:** Analyzes exact keyword frequency matching, penalized by how common the word is across the collection.
- **Failure Mode:** Lacks **conceptual context**. If a user asks *"How do I fix a broken session token?"*, but the compliance wiki uses *"Remediation steps for expired authentication credentials"*, BM25 scores zero because none of the exact words overlap.

---

## 2. The Production Standard: Hybrid Retrieval + RRF

Every user query executes two parallel retrieval vectors over the same dataset, merging their strengths.

```
                   ┌──► Dense Index (HNSW Graph) ──► Top 100 Candidates (Semantic) ──┐
User Query ────────┼                                                                 ├──► Reciprocal Rank Fusion (RRF) ──► Cross-Encoder Reranker ──► Context Top 5
                   └──► Sparse Index (BM25/SPLADE) ─► Top 100 Candidates (Keywords) ─┘
```

### Merging Scores: Reciprocal Rank Fusion (RRF)

Dense similarity scores (e.g., `0.84`) and BM25 scores (e.g., `24.5`) have entirely incompatible scale ranges. **RRF** ignores raw scores and uses only the *relative rank position* of an item within each result set.

$$RRF(d) = \sum_{m \in M} \frac{1}{k + r_m(d)}$$

Where:
- $r_m(d)$ is the rank of document $d$ in strategy $m$ (1-indexed).
- $k$ is a smoothing constant (standard production baseline is **60**).

If a chunk ranks #1 in BM25 (exact SKU match) but #80 in vector search, its integrated RRF score remains elevated, guaranteeing it bypasses the coarse filter stage.

## 3. High-Scale Implementation Models

Modern architectures do not require two separate systems (isolated Pinecone + standalone Elasticsearch), which introduces network latency and synchronization issues. Unified search kernels handle both in a single API call.

### Option A: Milvus 2.4+ Multi-Vector Paradigm

Milvus supports defining a collection with both a `FLOAT_VECTOR` (dense) field and a `SPARSE_VECTOR` (for SPLADE or BM25 tokens) in the same entity schema. It processes both indexes concurrently on GPU/CPU nodes and merges them inside the DB instance using an explicit `RRFRanker()`.

### Option B: Elasticsearch 8.x+ Search Retriever Paradigm

Elasticsearch handles this natively using the `retrievers` block, allowing a `knn` dense vector query and a standard text search query to run in parallel, passing immediately into an `rrf` processing layer before data leaves the database.

## 4. Architectural Traps

### Trap 1: Post-Filtering vs. Pre-Filtering for RBAC

- **Post-Filtering Mistake:** Running a vector search for the top 10 chunks, then stripping out chunks the user isn't allowed to see. If 9 out of 10 require admin rights, a standard employee receives only 1 chunk — the LLM gives a broken, incomplete response.
- **Production Rule:** Enforce **Pre-Filtering**. Pass the RBAC metadata array directly *into* the HNSW graph search criteria, so the database evaluates permission matches *during* index evaluation.

### Trap 2: HNSW Graph Memory Overhead

- **Cost Trap:** 1 million chunks with a 1536-dimension embedding model requires roughly **6 GB to 8 GB of pure RAM** just to keep the index active. Scale to 50 million chunks and infrastructure bills skyrocket.
- **Mitigation:** Configure **Scalar Quantization (SQ8 / int8)** inside vector index configurations. This compresses 32-bit floats into 8-bit integers, slashing RAM by **up to 75%** with virtually no retrieval accuracy drop.

### Trap 3: Re-ranking Bottleneck

- **Cost Trap:** Firing a Cross-Encoder model (e.g., Cohere Rerank v3) across 500 retrieved chunks per question chokes throughput and spikes latency past 1000 ms.
- **Production Fix:** Constrain hybrid database retrieval to a coarse filter max boundary ($K=50$ or $100$). Pass *only* those top candidates to the Re-ranker. The Re-ranker reduces them to the definitive top 5 chunks for the LLM prompt.

## 5. Pre-Retrieval Layer: Query Transformation & Expansion

Users write poor queries — typos, shorthand, vague questions. Embedding a query like *"token broken"* directly will match chunks literally containing "token" and "broken." Two algorithms fix this at the proxy layer before the database is touched:

### Multi-Query Expansion

- **Algorithm:** Route the user's string through a lightweight loop that generates $N$ alternative semantic variations.
- **Example:** User asks *"how to reset token"*, the expander outputs:
  1. *"OAuth session token expiration recovery procedure"*
  2. *"Steps to regenerate expired authentication credentials"*
  3. *"Handling invalid access token errors"*
- **Execution:** Parallel database lookups run for *all* variations. This drastically increases **Recall** (probability that the true answer block is captured).

### HyDE (Hypothetical Document Embeddings)

- **Algorithm:** Instead of searching with the user's question, the engine asks a fast language model to *generate a hypothetical ideal answer*.
- **Rationale:** Questions and answers live in different semantic spaces. A question looks like a request; an answer looks like a statement. HyDE transforms the question into a speculative *answer template*, embeds that fake answer, and searches the vector space.
- **Result:** Match scores jump significantly because an answer-shaped embedding matches against actual answer chunks stored in the database.

## 6. RBAC Integration at the Algorithm Level

The database — not the LLM — must filter data. When the user query is sent to the index, it must be accompanied by a logical expression array extracted from the verified security token (JWT):

```
                       [User Query Vector] + [User Roles: ["finance_team", "sg_employee"]]
                                                    │
                                                    ▼
                                    ┌──────────────────────────────┐
                                    │    Graph Traversal Engine    │
                                    └───────────────┬──────────────┘
                                                    │
                             Evaluates node metadata BEFORE advancing path
                                                    │
                             ┌──────────────────────┴──────────────────────┐
                             ▼                                             ▼
             [Node 104: Metadata matches role]             [Node 105: Role Mismatch]
                     Path Continues                              Path Dropped
```

### The Inverted Index Check

Modern engines run a **two-step single-stage scan**:

1. The database maintains a fast, compressed bitmap index (inverted index) of all chunk IDs associated with specific security tags (e.g., `role:finance_team`).
2. When the HNSW graph traversal begins, the routing path is bounded by this bitmap. The algorithm physically ignores graph connections leading to nodes that are bits-flipped to `0` (unauthorized) for that user's role mask.

## 7. Post-Retrieval Layer: Two-Stage Re-ranking

After hybrid search returns the top 100 chunks, sending all 100 to the LLM is too expensive and triggers the **"Lost in the Middle" phenomenon** (LLMs degrade in accuracy when buried in too much context). A **Two-Stage Retrieval Pipeline** reduces them to the best 5.

```
[All Data Chunks] ──► Stage 1: Coarse Filter (Bi-Encoder/BM25) ──► Top 100 Chunks
                                                                       │
                                                                       ▼
                      Stage 2: Fine Filter (Cross-Encoder Reranker) ──► Top 5 Chunks
```

### Stage 1: Bi-Encoder (Fast, Approximate)

- Embedding models process the document text once (during Phase 1 ingestion) and process the user query once at query time. Vectors are calculated independently.
- The database performs simple dot-product or cosine vector math.
- **Limitation:** Because the query and document never "talk" to each other inside the network, the model misses deep contextual interactions.

### Stage 2: Cross-Encoder (Slow, Ultra-Precise)

- Does not precompute anything. Takes the user's query and a document chunk, concatenates them into a single text block (`[CLS] User Query [SEP] Document Chunk [SEP]`), and feeds the *combined* text through a transformer network.
- Applies full, bidirectional attention across every word of the query and the chunk simultaneously. Spots subtle contradictions, context changes, or precise semantic overlaps that compressed vectors miss.
- **State-of-the-art models:** **Qwen3-Reranker-4B**, **NVIDIA nv-rerankqa**, **BGE-Reranker-Large**.

### Why Re-ranking Matters: A Concrete Example

User asks: **"Can I use my wife's Dependant's Pass to register a side project company in Singapore?"**

Standard hybrid search results (by raw vector distance):

1. **Doc A (Score 0.88):** *"How to apply for a Dependant's Pass (DP) for your spouse in Singapore."* — High score because words overlap heavily, but answers the *wrong question*.
2. **Doc B (Score 0.85):** *"Registering a private limited company microservice infrastructure setup."* — Matches business terms, misses immigration nuance.
3. **Doc C (Score 0.81):** *"Guidelines on DP holders starting a business via the LOC (Letter of Consent) framework."* — The **perfect answer**, ranked lowest because vocabulary is highly formal.

The Cross-Encoder re-evaluates the collection. It notices *"operating a business or side project"* perfectly satisfies the user's intent. **Doc C shoots to Rank 1**, while irrelevant documents drop.

### Re-ranking Code Example (Using `sentence-transformers`)

```python
from sentence_transformers import SentenceTransformer, CrossEncoder
from sklearn.metrics.pairwise import cosine_similarity
import numpy as np

# 1. Load both models
bi_encoder = SentenceTransformer('all-MiniLM-L6-v2')       # Stage 1: Fast Vector Search
cross_encoder = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2') # Stage 2: Deep Re-ranker

# 2. Database Documents
documents = [
    "How to apply for a Dependant's Pass (DP) for your spouse or wife in Singapore via the MOM portal.",
    "Setting up a private limited company software engineering project repository and CI/CD pipelines.",
    "MOM guidelines on Dependant's Pass (DP) holders operating a business or side project via an eligible Letter of Consent (LOC)."
]

query = "Can I use my wife's Dependant's Pass to register a side project company in Singapore?"

# =====================================================================
# STAGE 1: Fast Bi-Encoder / Vector Search
# =====================================================================
doc_embeddings = bi_encoder.encode(documents)
query_embedding = bi_encoder.encode([query])

initial_scores = cosine_similarity(query_embedding, doc_embeddings)[0]
stage1_ranking = np.argsort(initial_scores)[::-1]

print("--- Stage 1: Initial Vector Search Results ---")
for rank, idx in enumerate(stage1_ranking):
    print(f"Rank {rank+1} (Doc ID {idx}) | Score: {initial_scores[idx]:.4f} | Text: {documents[idx][:60]}...")

# =====================================================================
# STAGE 2: Deep Cross-Encoder Re-ranking (Only on the top results)
# =====================================================================
pairs = [[query, doc] for doc in documents]
rerank_scores = cross_encoder.predict(pairs)
stage2_ranking = np.argsort(rerank_scores)[::-1]

print("\n--- Stage 2: Re-ranked Results (The Correction!) ---")
for rank, idx in enumerate(stage2_ranking):
    print(f"Rank {rank+1} (Doc ID {idx}) | New Score: {rerank_scores[idx]:.4f} | Text: {documents[idx][:60]}...")
```

### Output Analysis

- **Stage 1 (Vector Search)** ranks Doc 0 highest simply because words like "Dependant's Pass", "spouse", "wife", and "Singapore" overlap heavily. The actual answer (Doc 2) is pushed to the bottom.
- **Stage 2 (Re-ranker)** notices "operating a business or side project" perfectly satisfies the user's intent. Doc 2 immediately shoots to Rank 1.

---

## Complete Retrieval Loop Summary

1. User types question.
2. Query Expander turns 1 question into 3 variations.
3. Hybrid Search runs those 3 queries, respecting the RBAC filter mask.
4. Top 100 chunks are returned within milliseconds.
5. Cross-Encoder ranks the 100 chunks and slices out the absolute top 5.

### Why Re-ranking is a Production Requirement

1. **Saves LLM Context Windows:** Instead of passing 20 poorly sorted documents (increasing token costs, causing the LLM to "lose" information in long prompts), pass just the top 3 highly accurate documents.
2. **Combines Multiple Sources Natively:** If Stage 1 pulls data from multiple places (10 chunks from a vector index, 10 from SQL keyword search, 10 from an issue tracker), throw all 30 into the Re-ranker for a clean, unified, universally scored list.
