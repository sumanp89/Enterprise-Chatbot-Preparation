# Phase 4: Production Guardrails & Automated Evaluation

With the processing layer (Phase 1), retrieval engine (Phase 2), and agent/tool execution architecture (Phase 3) built, this phase addresses how to stop the chatbot from causing damage when exposed to **10M peak users**.

A critical distinction:

- **Guardrails** are seatbelts — they run *live, in-production, on every single request* to block, modify, or approve text/actions.
- **Evaluations (Evals)** are crash tests — they run *offline, in CI/CD pipelines, before code hits production*, to measure quality and catch regressions.

---

## Part 1: The Production Guardrail Stack (4-Layer Defense)

A layered defense-in-depth model with a tight latency budget (~50 ms - 150 ms total overhead) to avoid destroying user experience.

```
                  ┌──► Layer 1: Inbound Input Scanner (Fast Filter / Regex)
                  │
User Interaction ─┼──► Layer 2: Core Safety Classifier (LlamaGuard 4 / Custom SLM)
                  │
                  ├──► Layer 3: Dynamic Tool Verification (API Gateway Firewall)
                  │
                  └──► Layer 4: Outbound Grounding Check (Guardrails AI / Self-RAG)
```

### Layer 1: Inbound Input Scanning (Lightweight Filter)

- **Task:** Catch the lowest-hanging fruit immediately before invoking expensive LLMs.
- **Mechanics:** High-speed pattern matching (regex) and small local tokenizers (like **LLM Guard**). Flags raw PII leaks (credit card numbers, social security patterns) and scans for basic known prompt-injection signatures.
- **Latency Cost:** < 10 ms.

### Layer 2: Deep Safety Classification (Intent Guard)

- **Task:** Detect advanced semantic jailbreak attempts (adversarial roleplay, system prompt override tricks) and toxic inputs.
- **Mechanics:** Route the user query through a specialized small safety classifier — such as Meta's **LlamaGuard 4 (12B)** or a localized DeBERTa model — running on internal GPU inference nodes.
- **Action:** The model outputs a binary `safe` or `unsafe` token. If `unsafe`, the orchestrator immediately blocks the request and short-circuits to a static response: *"I cannot assist with that request."*
- **Latency Cost:** ~30 ms - 50 ms.

### Layer 3: Tool & Action Gating

*(Locked down in Phase 3 — the API Gateway ensures parameter structure and verifies the user's JWT roles before executing any microservice.)*

### Layer 4: Outbound Grounding & Hallucination Defense

- **Task:** Inspect the LLM's draft response *before* it leaves the network to ensure it hasn't hallucinated facts or leaked internal data.
- **Mechanics:** Structural validation rules (like **Guardrails AI** or programmatic **Self-RAG** checks) map the generated answer against the top 5 retrieved context chunks from Phase 2.
- **Rule:** If the model introduces a factual claim or numeric metric that does *not* exist inside the text chunks, the guardrail triggers a fallback or automatically rewrites the response to prevent hallucination.
- **Latency Cost:** ~40 ms - 80 ms.

## Part 2: The Evaluation (Eval) Framework

An **Automated Evaluation Pipeline** runs inside GitHub Actions / GitLab CI/CD pipelines to prove that code or prompt changes won't break the system before shipping to 10M users.

```
                           [CI/CD Code Push]
                                   │
                                   ▼
         [Run Synthetic Test Suite (1,000+ Adversarial Prompts)]
                                   │
                                   ▼
                     [LLM-as-a-Judge Evaluation]
                                   │
          ┌────────────────────────┴────────────────────────┐
          ▼                                                 ▼
   [Score < Threshold]                               [Score Passes]
    Block Deployment                                  Deploy to Production
```

### Core Metrics

An LLM-as-a-Judge matrix (using open-source engines like **Ragas** or **Phoenix**) grades responses across three atomic metrics:

- **Faithfulness / Grounding:** Measures if the generated response is *strictly* derived from retrieved documents. (Score: 0.0 to 1.0).
- **Answer Relevance:** Measures if the output actually answers the user's specific problem, or drifted off-topic.
- **Context Precision:** Evaluates if the Phase 2 RAG pipeline retrieved *only* relevant information, or introduced noisy unneeded chunks.

### Adversarial Dataset (Red Team)

A static golden dataset of over 1,000 synthetic test cases including complex edge cases, multi-turn trick queries, and historical prompt-injection payloads. If a developer edits a system prompt, the CI/CD pipeline runs this entire dataset. If the overall safety score drops by even **0.5%**, deployment is automatically blocked.

## Part 3: Outbound Grounding & Hallucination Defense — Deep Dive

If a document chunk says the refund policy is 30 days, but the model hallucinates "60 days," the enterprise is exposed to operational and legal risk. The industry has standardized around specific frameworks and models built to detect hallucinations within milliseconds.

### Tier 1: Open-Source Structural Guardrail Frameworks (Code Enforcers)

These wrap around LLM API calls. They define strict programmatic constraints or validation hooks that run the moment a model outputs text.

- **Guardrails AI:** Industry-standard framework using a `.rail` schema configuration file. Maps the LLM's output against validation scripts (e.g., verifying every entity in the response exists in the source text array). On failure, it can trigger a localized re-prompt loop or block the message.
- **NVIDIA NeMo Guardrails:** Uses a modeling language called **Colang** to create an execution sandbox. Rails define precise flow states (e.g., *"If the model output contains facts not present in the RAG context variable, execute the fallback-message flow"*).

### Tier 2: Specialized NLI Models (Math Engines)

Instead of using a heavy commercial LLM to check for hallucinations, pipe the generated answer and source chunks through a tiny, hyper-fast local **NLI Classification Model** on internal GPU nodes.

- **Vectara HHEM-2.1 (Hughes Hallucination Evaluation Model):** Open-weights model engineered exclusively for RAG grounding verification.
  - **Mechanics:** Treats the RAG context as the "Premise" and the LLM's draft answer as the "Hypothesis." Runs a single binary classification pass, outputting a probability score from 0.0 to 1.0 indicating how well the answer is supported by the source text.
  - **Performance:** Executes in **< 30 ms** on standard cluster nodes — invisible to the user's total latency window.
- **Granite-Guardian (IBM Research) / LlamaGuard 4 (12B):** Small open-source models with specialized safety tuning. In "grounding mode," they act as zero-shot fact checkers, outputting a tag specifying whether the output drifts into unverified territory.

### Tier 3: Managed Enterprise Security Gateways (Plug-and-Play)

For teams choosing to offload infrastructure management entirely, LLM traffic routes through a specialized proxy layer like **Javelin AI Gateway** or **Portkey**. These platforms embed real-time pre- and post-processing filters directly inside their proxy networks, running automated semantic grounding checks before data reaches the client application.

## Part 4: The DIY Production Pattern — Self-RAG

To avoid vendor lock-in, an **Algorithmic Verification Loop (Self-RAG)** can be implemented directly within the orchestration worker code. Instead of hoping the model is right, the orchestration loop explicitly verifies the output step-by-step:

```
[LLM Draft Output] ──► Task 1: Check Context Sufficiency (Is there enough info?)
                                │
                                ▼
                       Task 2: Check Factual Grounding (Are there fake claims?)
                                │
                                ▼
                       Task 3: Check Answer Utility (Does it solve the user's issue?)
```

### Programmatic Control Loop

Implemented using a fast, quantized local model (Mistral or Llama-3-8B) executing structured JSON parsing instructions:

```python
def verify_outbound_payload(retrieved_chunks, model_draft_response):
    verification_payload = f"""
    Given the following reference context, determine if the response contains any facts, 
    numbers, or assertions that are not explicitly stated in the context.
    
    Reference Context: {retrieved_chunks}
    Draft Response: {model_draft_response}
    
    Respond strictly in JSON format matching this schema:
    {{
      "is_grounded": boolean,
      "unverified_claims": ["array of strings or empty"]
    }}
    """
    
    validation_output = call_internal_fast_llm(verification_payload)
    
    if not validation_output.is_grounded:
        log_system_hallucination_alert()
        return "I've processed your request, but I cannot verify the exact metrics in our records right now. Please let me connect you with a representative."
        
    return model_draft_response
```

## Recommendation for 10M Peak Users

**Do not use a large commercial LLM to check another LLM's work.** It doubles token costs and ruins latency metrics.

The optimal architecture combines **Guardrails AI** to orchestrate the validation structure, paired with a local self-hosted deployment of **Vectara's HHEM model** running on internal cluster infrastructure. This ensures every sentence is mathematically checked for factuality against source documents within a tight **30 ms** processing window, maintaining security and ultra-fast user experiences.
