# Phase 3.2: Multi-Turn Session State & Memory Architecture

At a scale of **1M steady to 10M peak daily users**, passing long raw chat histories back and forth to the LLM quickly breaks down.

---

## 1. Core Scaling Failures of Conversational Memory

If a user chat lasts 20 to 30 turns, storing and passing every message as raw text creates three runtime bottlenecks:

- **Latency and Cost Snowball:** If each message exchange averages 200 tokens, by turn 20 the conversation history alone consumes **4,000 tokens**. Every new message re-transmits those 4,000 tokens, causing input costs and Time-To-First-Token (TTFT) latency to spike higher with each turn.
- **Context Limit Wall:** An agentic workflow looping through multiple documents and tools can hit the LLM's context limit, causing the application to crash or drop older conversations mid-session.
- **Database IOPS Exhaustion:** If the orchestration layer performs heavy, unindexed read/write operations against a standard database (PostgreSQL or disk-heavy MongoDB) to fetch the full chat history for *every single user turn*, database connections exhaust during traffic spikes.

---

## 2. Dual-Layer Tiered Memory Architecture

Session memory is split into two decoupled, high-speed storage tiers to handle millions of parallel sessions without degrading response times.

```
                      ┌──► Active RAM (Redis Cluster) ──► Short-Term Sliding Window (Last N Turns)
                      │
User Session ID ──────┼──► Cold/Warm Storage (DynamoDB) ─► Complete Append-Only Chat History Log
                      │
                      └──► Async Background Worker ─────► Periodic Summarization (Updates Context Paragraph)
```

### Tier A: Short-Term Sliding Window (Active Memory)

- **Infrastructure:** Enterprise **Redis Cluster** with replication and sharding. Reads and writes execute in **< 2 ms**, protecting API threads from blocking.
- **Content:** Only the last **3 to 5 raw turns** stored in a fast-expiring JSON or List structure with a strict TTL set to 1 hour. Provides enough immediate context for the LLM to understand references like *"What did it say about the second option?"*.

### Tier B: Long-Term Archive (System of Record)

- **Infrastructure:** Key-value store optimized for high-write velocity — **Amazon DynamoDB** or **GCP Bigtable**.
- **Content:** The complete, unedited history of every turn, streamed asynchronously. Serves as the audit trail for data compliance and analytics, completely isolated from the fast search path.

---

## 3. The Compaction Loop: Asynchronous Rolling Summarization

To maintain long-term context across multi-hour or multi-day interactions without bloating the runtime prompt, an asymmetric background worker pattern compresses the history.

```
[Raw Turns 1 to 5] ──► Move to Tier B ──► Trigger Background LLM ──► Save "Semantic Summary Context"
                                                                                 │
                                                                                 ▼
                                                                  Prepended to Turn 6 Prompt
```

1. **Window Threshold:** As the conversation crosses a set message limit (e.g., turn 6), the orchestrator strips the oldest turns (turns 1 and 2) out of the active Redis window.
2. **Async Summary Task:** The orchestrator emits a lightweight event to a background processing queue (Kinesis or Pub/Sub). A consumer worker grabs the historical log and passes it through a compressed, fast LLM call with the instruction: *"Condense this conversation into a single paragraph tracking key user decisions, extracted details, and current system states."*
3. **Summary Merge:** The output string updates a dedicated `"semantic_summary"` attribute field in Redis for that Session ID.
4. **Final Assembled Context:** When the user submits turn 7, the prompt assembly engine builds the input payload using compact building blocks:

```xml
<conversation_summary>
User is checking Singapore account compliance guidelines for lockouts. 
Verified internal admin role. System token reset was initiated but failed on step 2.
</conversation_summary>

<recent_dialogue_window>
User: "Why did step 2 fail?"
Assistant: "The proxy layer returned a timeout error while reaching the authentication microservice."
User: "Can you re-run it now?"
</recent_dialogue_window>
```

---

## 4. Architectural Traps to Avoid

- **Atomic Session Locks (Race Condition Prevention):** With millions of users, a user might open multiple tabs or hit a button repeatedly, causing 2-3 requests from the same session ID to hit the API gateway simultaneously. If multiple nodes try to read, modify, and write to the same session memory key at the same millisecond, they overwrite each other's states. An in-memory **Distributed Lock (Redlock)** via Redis must be enforced during session modification to ensure requests for the same session ID are processed sequentially.
- **State Hydration Offloading:** Never pass session parsing logic through the primary generation model. The orchestration worker should handle fetching keys, computing sliding array math, and assembling the string container before the payload hits the LLM execution node.
