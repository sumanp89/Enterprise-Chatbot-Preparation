# Vector Database Selection: Architecture & Cloud Provider Analysis

Selecting the vector storage layer for an enterprise application handling **1M baseline scaling to 10M peak daily users**. This document covers standalone database candidates, cloud-managed offerings, and the key architectural trade-offs.

---

## Part 1: Standalone Database Candidates

### 1. Milvus (Distributed Open-Source / Zilliz Cloud)

Milvus is built as a distributed microservices system, completely decoupling **Compute** from **Storage**.

- **Architecture:** Data is broken into segments. **Query Nodes** handle memory-resident search queries, **Index Nodes** build vector graphs on separate workers, and **Data Nodes** stream raw blocks to object storage like S3/GCS.
- **Multi-Tenancy:** Implements partitions and **dynamic metadata fields**. When an RBAC token is passed, Milvus uses an inverted index on scalar columns to prune data segments *before* searching the vector index graph.
- **Trade-off:** Operating self-managed requires a Kubernetes cluster, an Etcd coordination layer, and MinIO/S3 storage. The operational footprint is heavy, but the scaling ceiling is virtually limitless.

### 2. Elasticsearch 8.x / OpenSearch

Elasticsearch treats vectors as specialized properties inside standard JSON documents, powered by Apache Lucene.

- **Architecture:** Lucene builds **HNSW (Hierarchical Navigable Small World)** graphs directly inside native segments alongside its traditional inverted text indexes (BM25).
- **Hybrid Search:** Because the dense vector and text tokens live inside the exact same Lucene segment on disk, the database executes a text query and a vector query simultaneously against the same block of data, eliminating network serialization overhead.
- **Trade-off:** Lucene requires the entire HNSW graph to reside in off-heap RAM for optimal performance. If indices outgrow the RAM allocation, performance drops dramatically as pages swap to disk.

### 3. pgvector (PostgreSQL Extension)

pgvector introduces a custom vector data type (`vector(dimensions)`) and custom table index structures into standard PostgreSQL.

- **Architecture:** `CREATE INDEX ON table USING hnsw` constructs an internal graph network directly inside standard table pages.
- **Relational Joins:** The greatest superpower — user data, RBAC permissions, and vector chunks live inside standard relational tables. An RBAC check is just a fast SQL `WHERE` clause join.
- **Trade-off:** At scale (millions of users, tens of millions of chunks), building or rebuilding an HNSW index locks up database workers and can take hours. Vacuum processes can choke high-throughput stream ingestion.

### 4. Pinecone (Serverless Managed Vector Cloud)

Pinecone is a closed-source, purpose-built, fully managed serverless vector environment.

- **Architecture:** Storage and compute are separated. Vectors and metadata payloads are stored permanently in flash memory/SSD. When a query fires, it dynamically pulls the required graph indices into transient compute nodes.
- **Multi-Tenancy:** Uses explicit `namespaces`. Different user groupings or tenants are isolated into separated logical containers — a query in Namespace A can physically never see data in Namespace B.
- **Trade-off:** Zero control over underlying indexing parameters (e.g., tuning graph construction density), and text indexing is secondary to vector indexing.

## Part 2: Cloud-Managed Vector Offerings

### Amazon Web Services (AWS)

AWS provides two main avenues for production vector search.

#### A. Amazon OpenSearch Service

- **Mechanics:** Uses a specialized `k-NN` plugin. It can run in two modes: **Lucene-based HNSW** or **NMSLIB-based HNSW**.
- **Benefit:** Fundamentally a text search engine, so it handles **Hybrid Search natively** — BM25 keyword query and dense vector `k-NN` query execute in a single execution block, with results merged seamlessly.
- **Limit:** Highly RAM-dependent. The HNSW graph must reside in memory. For tens of millions of vectors, memory-optimized instances (`r6g` or `r7g`) will scale the monthly baseline cost significantly.

#### B. Amazon S3 Vectors

- **Mechanics:** A serverless, object-storage-native vector bucket structure. Builds vector indexes directly on top of raw S3 buckets, handling up to **2 billion vectors per index**.
- **Benefit:** Drops storage and upload costs by up to **90%** compared to provisioned compute nodes. Ideal for elastic serverless billing when the data corpus scales massively.
- **Limit:** Latency — roughly **100 ms** for warm queries due to the object storage architecture. Works better as a massive background knowledge index rather than a high-speed transactional memory layer if sub-50ms response times are required.

### Google Cloud Platform (GCP)

GCP focuses on localized database acceleration and massive scale graph analytics.

#### A. AlloyDB AI

- **Mechanics:** GCP's highly optimized PostgreSQL-compatible database. Instead of standard `pgvector` HNSW indexes, AlloyDB AI integrates **Google's proprietary ScaNN (Scalable Nearest Neighbors) index**.
- **Benefit:** ScaNN uses anisotropic vector quantization to compress vectors tightly and executes graph lookups **up to 6x faster** than standard PostgreSQL. Handles up to **10 billion vectors** while retaining full relational SQL capability — RBAC user tables and vector data reside in the same transactional machine.
- **Limit:** Requires commitment to the AlloyDB infrastructure envelope, which carries a higher base operating cost than standard Cloud SQL instances.

#### B. Vertex AI Vector Search

- **Mechanics:** Formerly known as Matching Engine. A fully managed, purpose-built vector graph deployment engine based on Google's internal search algorithms.
- **Benefit:** Built for raw scale — handles billions of vectors with sub-10ms latencies at extreme QPS. Shards indexes across decentralized compute blocks automatically.
- **Limit:** Strictly a vector index. Does **not** store raw text strings or documents — only vector coordinate arrays and an ID. The application must fetch actual text content from another store (Cloud Bigtable, Firestore), adding a secondary network hop to the retrieval loop.

### Microsoft Azure

Azure's core strength is a highly unified, turnkey retrieval platform for enterprise RAG workflows.

#### A. Azure AI Search

- **Mechanics:** A fully managed enterprise search orchestration engine. Embeds vector indexing (HNSW), native BM25 text keyword matching, and integrated semantic ranking pipelines into a single managed SaaS endpoint.
- **Benefit:** Features **Integrated Two-Stage Re-ranking** out-of-the-box — executes a coarse hybrid filter pass, then pipes top results through a built-in Microsoft Turing cross-encoder re-ranker before delivering text to the application. Saves building a custom re-ranking microservice.
- **Limit:** Closed platform wrapper. Very limited ability to tune low-level index parameters (like graph connectivity metrics), and operates entirely outside standard relational database topologies.

---

## Alignment Assessment (AWS + GCP)

Given the deployment flexibility across **AWS** and **GCP**, the choice narrows to two distinct architectural philosophies:

1. **Distributed Search Route (AWS OpenSearch):** Standard keyword + vector hybrid control in one tool. Requires managing memory clusters.
2. **Relational Acceleration Route (GCP AlloyDB AI):** Keeps all RBAC logic and data in standard Postgres tables, uses Google's ScaNN engine for massive throughput.
