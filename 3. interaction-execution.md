# Phase 3: The Interaction & Execution Layer (Agent Orchestration & Core Generation)

At this point the matching data shards are retrieved, but they haven't been given to the LLM yet. This layer also handles user **Persona** alignment and the intent to execute live backend microservices.

---

## Phase 3 Overview

At a scale of millions of users, this layer splits into four atomic building blocks that run sequentially:

```
                  ┌──► Informational Route ──► Prompt Context Assembly ──► LLM Generation
                  │
[Top 5 Chunks] ───┼──► Transactional Route ──► State Machine Evaluation ──► Multi-Turn Tool Execution
                  │
                  └──► Persona Alignment ─────► Dynamic System Prompt Injection
```

### 1. Dynamic System Prompt Assembly

A single giant system prompt for every query is an anti-pattern. Loading an LLM with rules for IT Support, Billing Transactions, and General FAQ simultaneously balloons prompt context size (increasing token costs), and the model experiences attention drift, confusing persona roles.

- **Design:** A **State-Driven Router** analyzes user context before hitting the generation model via a fast, lightweight classification step.
- **Execution:** If the user is flagged as an `internal_admin` looking at compliance documentation, the system dynamically fetches the **Admin Persona Prompt Fragment** from a fast cache (Redis) and prepends it to the LLM instruction set, ignoring consumer customer rules entirely.

### 2. Multi-Turn Session State & Memory Architecture

Passing the entire raw history of a 20-turn chat to the LLM destroys performance, spikes input token costs exponentially, and hits context limits during peak traffic.

- **Design:** Conversation memory splits into two structural layers:
  - **Transient Window:** Only the last 3 raw conversation turns in active memory for fast context follow-ups.
  - **Rolling Summary Store:** For turns 4 through 20, a background worker asynchronously condenses older chat history into a dense semantic paragraph summary. The generation node reads: `[Rolling Summary] + [Last 3 Raw Turns] + [Top 5 RAG Chunks]`.

### 3. Actionability & Tool Call (Function Calling) Safeguards

The application is **agentic** — it calls REST APIs, processes user transactions, and triggers downstream flows. If an LLM decides to call a backend billing endpoint based on a user request, core databases must be protected from bad tool parameters or prompt injections.

- **Design:** The LLM does *not* execute code directly. It only outputs a structured intent block (a JSON schema declaring a function name and arguments).
- **Execution:** The JSON payload lands in an application gateway proxy. The gateway runs strict **Deterministic Type Validation** and input parameter checks *before* routing the call to the internal microservice network.
