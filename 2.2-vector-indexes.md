# Vector Index Types: Mechanics, Compression & Production Configuration

At a scale of **10 million peak users**, the vector database will hold millions of chunk vectors. If the underlying index is not deliberately chosen, the database will either **OOM crash**, **take 500 ms to respond (latency death)**, or **return completely irrelevant results (low recall)**.

This document dissects the core vector index families, their math and mechanics, compression via quantization, and the production implications of each choice.

---

## The Three Structural Index Families

Finding a matching vector means locating the closest points in a high-dimensional space (e.g., 1536 dimensions). Scanning every vector one by one is too slow. Indexes group vectors using different structural strategies to avoid brute-force scans.

### 1. Graph-Based Index: HNSW (Hierarchical Navigable Small World)

HNSW is the current gold standard for in-memory speed. It functions as a multi-layer skip-list built out of graph networks rather than linear chains.

```
Layer 2 (Express Lane):    [Node A] ───────────────────────────────► [Node M]
                             │                                         │
Layer 1 (Local Lane):      [Node A] ─────────► [Node F] ─────────────► [Node M]
                             │                   │                     │
Layer 0 (All Vectors):     [Node A] ─► [Node B] ─► [Node F] ─► [Node H] ─► [Node M]
```

- **Layer Hierarchy:** The top layer has very few nodes with long-range connections. Search starts here, making massive spatial jumps. As it nears the target area, it drops down a layer to inspect closer, more tightly bound local connections.
- **Graph Construction:** When a new vector is inserted, the engine searches the graph to find its $M$ closest neighbors and draws permanent edges to connect them.
- **Trade-off at 10M Users:** To navigate these nodes at sub-10ms speeds, **the entire graph structure must live in RAM.** Every vector node stores a list of memory addresses for all its connected neighbors, so the index graph can grow to **2x to 5x the size of the raw vector data.**

### 2. Clustering-Based Index: IVF (Inverted File Index)

IVF relies on classical multi-dimensional clustering ($k$-means) to partition the vector space into distinct geometric regions called Voronoi cells.

```
                 ● Vector    ★ Centroid (Cluster Center)
          ┌─────────────────┬─────────────────┐
          │     ●     ●     │    ●       ●    │
          │        ★        │        ★        │
          │   ●       ●     │    ●       ●    │
          └─────────────────┴─────────────────┘
                Cell 1            Cell 2
```

- **Centroid Training:** During an upfront training phase, the database evaluates the dataset and picks $N$ central points called "centroids". Every vector is assigned to its nearest centroid bucket.
- **Probe Query:** When a user queries, the database finds the 1 or 2 closest *centroids* (`nprobe`), unlocks only those specific buckets, and scans the vectors inside them.
- **Trade-off at 10M Users:** IVF uses very little memory (~1.1x raw vector size). However, **it handles continuous live data writes poorly.** If Near-Real-Time or 6-Hour pipelines push thousands of new vectors, the cluster cell shapes shift out of alignment over time. Search recall drifts downwards until an expensive global re-clustering training job is run.

### 3. Disk-Backed Graph Index: DiskANN / Vamana

Developed by Microsoft Research, DiskANN was created specifically to break the memory wall of graph search.

- **Architecture:** Uses a specialized graph structure called **Vamana**, which has shorter, tighter search paths than HNSW.
- **The Disk Shift:** Compresses coordinate representations kept in RAM into tiny structural tokens, but stores full high-fidelity geometric graphs directly onto ultra-fast NVMe SSD disks. Queries resolve initial coordinates in memory, then make highly concurrent asynchronous I/O reads against the SSD for final matching vectors.
- **Trade-off at 10M Users:** Drops RAM costs by **up to 10x** — billions of vectors can live on cheap disk blocks instead of expensive memory. The trade-off is tail latency: DiskANN lookup takes roughly **20 ms to 40 ms**, whereas HNSW responds in **2 ms to 5 ms**.

## The Compression Layer: Quantization (SQ8 vs. PQ)

Before storing vectors inside any index family, mathematical compression can be applied to the coordinate arrays to shrink the memory footprint.

### Scalar Quantization (SQ8)

Converts standard 32-bit floating-point numbers (`float32`) into 8-bit signed integers (`int8`). Truncates decimal precision slightly but drops memory usage by **exactly 75%** with less than a 1% drop in search recall.

### Product Quantization (PQ)

Slices a high-dimensional vector (e.g., 1024 dimensions) into 16 smaller sub-vectors, maps them against a localized codebook, and replaces the raw numbers with small byte coordinate IDs. Can compress the index by **over 90%**, but introduces lossy compression errors that can degrade accuracy if data contains highly nuanced semantic differences.

## Why Index Choice Dictates Architecture

If index configurations are not selected deliberately during database setup (whether using AWS OpenSearch, GCP AlloyDB, or Milvus), the engine falls back to defaults. At production scale, this is unacceptable.

### RBAC Latency Collapse: Pre-Filtering Strategy

Millions of users query data restricted by strict RBAC roles.

- **IVF Index + RBAC Filter:** The database scans matching cells. If a premium user's role metadata matches only 2% of the documents in the cluster buckets, the database wastes cycles evaluating thousands of rows it must throw away.
- **HNSW Index + RBAC Filter:** The database tracks payload metadata attributes inside graph nodes. As it jumps from node to node across layers, it evaluates the boolean RBAC filter *at the edge level*. If a path leads to unauthorized data, the traversal engine ignores that entire graph leg, preserving sub-10ms response times even under highly restrictive roles.

### Sizing Compute Nodes

Raw math for a 10M peak daily user corpus. Assume 20 million text chunks across all enterprise wikis, PDFs, and HTML inputs, using a 1,536-dimension embedding model:

- **Unindexed/Flat Storage:** ~122 GB of static memory data.
- **Standard HNSW Index** ($M=16$, $ef=64$): Memory requirements balloon to **over 300 GB of pure RAM** for the index. Without quantization, this requires a massive cluster of high-tier memory instances with expensive baseline cost.
- **HNSW + SQ8 Quantization:** Compresses the same 20-million chunk graph down to **under 80 GB of RAM**, runnable on standard, lower-cost infrastructure nodes without sacrificing response speed.

### Recommended Index Configuration

Given the requirements for continuous updates (from 6-Hour and Near-Real-Time lanes) and ultra-low latency for millions of users, **HNSW + Scalar Quantization (SQ8)** is the optimal configuration. It protects search recall from drifting during active writes and cuts RAM infrastructure costs significantly.

## HNSW Deep Dive

HNSW is used by Pinecone, Milvus, and Qdrant for **Approximate Nearest Neighbor (ANN)** searches. Instead of comparing a query vector against billions of rows one by one (Brute Force / $O(N)$), HNSW drops search time to **logarithmic time ($O(\log N)$)** via a multi-layered graph network.

### Real-World Analogy

Think of HNSW like navigating a highway system:

- **Layer 2 (Interstate Highway):** Travel hundreds of miles across states using long, sparse exits. Incredibly fast, but only hit major hubs.
- **Layer 1 (State Expressways):** Exit onto regional expressways to reach the correct city. Tighter roads, more exits.
- **Layer 0 (Residential Roads):** Local streets. Every house is connected. Crawl slowly until parked at the exact address.

### Worked Example

Vector database stores simple 2D coordinates. Query: **Find the closest vector to the target point `[48, 50]`**.

```
[Layer 2: Interstate]     (Entry Point: New York) --------------------> (Chicago)
                               |                                            |
[Layer 1: Expressway]     (New York) -> (Philadelphia) -> (Cleveland) -> (Chicago)
                               |                                            |
[Layer 0: Local Streets]  (NY) -> (Trenton) -> (Philly) -> (Pitt) -> (Cle) -> (Chi)
```

**Search Steps:**

1. **Layer 2:** Algorithm drops at entry point (New York). Checks neighbors. Chicago is mathematically closer to target `[48, 50]`. Takes the long-range jump to Chicago.
2. **Drop Down:** At Chicago, no closer neighbors exist on Layer 2. Drops to Chicago on **Layer 1**.
3. **Layer 1:** From Chicago, evaluates closer regional neighbors. Cleveland is slightly closer to the target. Hops to Cleveland. Stalls.
4. **Layer 0:** Drops to Cleveland on the dense base layer containing *all* data. Makes micro-hops along local streets until standing on the absolute closest neighbor.

By using upper layers to "skip" past thousands of irrelevant vectors, only a tiny handful of distance calculations are needed.

### Python Code Implementation (Using `faiss`)

```python
import numpy as np
import faiss

# 1. Setup Dummy Data (10,000 vectors, each with 64 dimensions)
dimensions = 64
n_documents = 10000

np.random.seed(42)
database_vectors = np.random.random((n_documents, dimensions)).astype('float32')

# 2. Configure HNSW Parameters
# M = Max number of connection edges a node can have to its neighbors
M = 16  

# 3. Instantiate the HNSW Index 
# IndexHNSWFlat creates an HNSW graph structure that stores raw, uncompressed vectors
index = faiss.IndexHNSWFlat(dimensions, M)

# Optional: Tune accuracy vs speed while building
# efConstruction dictates how many neighbors to evaluate when inserting new vectors
index.hnsw.efConstruction = 64 

# 4. Build the Graph (Adds vectors and probabilistically builds the multi-layer lanes)
index.add(database_vectors)
print(f"Total vectors successfully indexed in HNSW graph: {index.ntotal}")

# 5. Execute a Vector Search Query
query_vector = np.random.random((1, dimensions)).astype('float32')

# efSearch dictates how many paths to explore down on Layer 0 during query runtime
index.hnsw.efSearch = 32
k_nearest_neighbors = 5

# Perform search: returns matrix of distances and corresponding matrix of row IDs
distances, indices = index.search(query_vector, k_nearest_neighbors)

print("\n--- HNSW Nearest Neighbor Search Results ---")
for rank, (idx, dist) in enumerate(zip(indices[0], distances[0])):
    print(f"Rank {rank+1} | Vector Row ID: {idx} | L2 Distance Score: {dist:.4f}")
```

### Key HNSW Parameters

- **`M`**: Controls graph density. Higher `M` = more accurate paths (better recall), but significantly more RAM. Typical production values: `16` to `64`.
- **`efConstruction`**: Controls build-time effort. Higher values make the DB work harder during data upload to find ideal highway links.
- **`efSearch`**: Controls query-time accuracy. Higher values expand the dynamic candidate search pool on Layer 0. If the RAG system is missing obvious answers, bump `efSearch` up to trade a tiny bit of latency for higher search accuracy.

## SQ8 (Scalar Quantization) Deep Dive

### The Memory Problem

By default, modern AI embedding models output vectors as arrays of **32-bit floating-point numbers (`float32`)**.

- Each `float32` number takes up **4 bytes** of memory.
- Using a 1,536-dimension model, a single vector: $1,536 \times 4 \text{ bytes} = 6,144 \text{ bytes} \approx 6 \text{ KB}$
- At **10 million vectors**: ~60 GB of raw RAM just to keep the application alive.

**SQ8 transforms every 4-byte `float32` into a 1-byte `int8`, cutting memory by exactly 75%** (down to ~15 GB for 10 million vectors).

### How SQ8 Works: Min-Max Uniform Quantization

An `int8` type can represent $2^8 = 256$ distinct values. For signed integers, the range is **-128 to +127**. The database squeezes continuous decimals into rigid integer buckets.

#### Setup Data

```python
original_vector = [0.1523, -0.8941, 0.4119] # Raw Float32 values
```

#### Step 1: Scan Dataset for Limits

The database scans the global dataset to identify boundaries:
- **Global Min** = $-1.0$
- **Global Max** = $+1.0$
- Total range = $2.0$

#### Step 2: Calculate Scaling Factor ($\alpha$)

Map the continuous range ($2.0$) onto 256 discrete integer steps (-128 to +127):

$$\alpha = \frac{\text{Global Max} - \text{Global Min}}{255} = \frac{1.0 - (-1.0)}{255} = \frac{2.0}{255} \approx 0.007843$$

#### Step 3: Quantize (Compress)

Convert raw float $v$ to `int8` counterpart $q$:

$$q = \text{round}\left( \frac{v - \text{Global Min}}{\alpha} \right) - 128$$

Applied to each dimension:

1. **Dimension 1 (`0.1523`):**

$$q_1 = \text{round}\left(\frac{0.1523 - (-1.0)}{0.007843}\right) - 128 = \text{round}(146.92) - 128 = 147 - 128 = \mathbf{19}$$

2. **Dimension 2 (`-0.8941`):**

$$q_2 = \text{round}\left(\frac{-0.8941 - (-1.0)}{0.007843}\right) - 128 = \text{round}(13.50) - 128 = 14 - 128 = \mathbf{-114}$$

3. **Dimension 3 (`0.4119`):**

$$q_3 = \text{round}\left(\frac{0.4119 - (-1.0)}{0.007843}\right) - 128 = \text{round}(180.02) - 128 = 180 - 128 = \mathbf{52}$$

#### Result in Memory

```python
quantized_vector = [19, -114, 52] # Saved as int8
```

Instead of **12 bytes** ($3 \times 4$ bytes), it now takes **3 bytes** ($3 \times 1$ byte).

### Search Execution: Dequantization

During search, the database converts the query using the same scaling factor. When traveling down the HNSW graph, it reads `int8` values and performs dequantization:

$$v_{approx} = \text{Global Min} + (q + 128) \times \alpha$$

Reconstructing dimension 1:

$$v_{approx} = -1.0 + (19 + 128) \times 0.007843 = -1.0 + 1.1529 = \mathbf{0.1529}$$

The compressed value (`0.1529`) is extremely close to the original (`0.1523`). The truncation error is minor enough that distance calculations remain highly accurate.

### Production Impact

- **RAM Footprint:** Cut by exactly **75%**. Server overhead costs drop linearly.
- **CPU Acceleration:** Modern CPUs (and GPUs) execute integer arithmetic (`int8`) significantly faster than floating-point math (`float32`).
- **Recall Drop:** Industry benchmarks (HotpotQA, MS-MARCO) show switching from `float32` to SQ8 drops Recall@10 by an almost imperceptible **0.5% to 1.4%**.

For nearly every enterprise application, losing 1% accuracy to save 75% on infrastructure costs is an immediate win.
