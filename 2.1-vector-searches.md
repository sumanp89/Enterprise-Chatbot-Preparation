# Types of Vector Search: Paradigms, Implementations & Architecture

Vector search moves information retrieval from rigid keyword matching to understanding actual intent and context. This document covers the **four primary search paradigms**, their Python implementations, the evolution from split to unified database architectures, and the distinction between traditional and AI-powered lexical search.

---

## 1. Pure Dense Vector Search (Semantic)

Transforms text into a continuous vector of numbers (usually 384 to 1536 dimensions) using Deep Learning models like BERT. Captures the **meaning, context, and synonyms** of the text rather than the exact words.

- **Pros:** Excellent at understanding intent, handling synonyms ("couch" vs. "sofa"), and multilingual queries.
- **Cons:** Can miss exact serial numbers, rare product IDs, or highly specific jargon; computationally expensive.

### Code Implementation

```python
from sentence_transformers import SentenceTransformer
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity

# 1. Load a lightweight, high-performing dense model
model = SentenceTransformer('all-MiniLM-L6-v2')

# 2. Sample Documents (The Database)
documents = [
    "The quick brown fox jumps over the lazy dog.",
    "Artificial intelligence and machine learning are transforming industries.",
    "A warm cup of coffee in the morning makes me happy.",
    "Deep neural networks are a subset of AI technologies."
]

# 3. Generate Dense Embeddings (Vectors filled with continuous floats)
doc_embeddings = model.encode(documents)
print(f"Embedding shape: {doc_embeddings.shape}")  # Output: (4, 384)

# 4. Define a Query (Notice it uses synonyms like 'Neural' and 'Data Science')
query = "What is the impact of neural models in data science?"
query_embedding = model.encode([query])

# 5. Calculate Similarity Score (Dense Vector Search)
# cosine_similarity expects 2D arrays, returns a matrix of scores
scores = cosine_similarity(query_embedding, doc_embeddings)[0]

# 6. Rank and Print Results
ranked_indices = np.argsort(scores)[::-1]

print("\n--- Dense Search Results ---")
for idx in ranked_indices:
    print(f"Score: {scores[idx]:.4f} | Doc: {documents[idx]}")
```

**Example Output:**

```
Embedding shape: (4, 384)

--- Dense Search Results ---
Score: 0.6133 | Doc: Deep neural networks are a subset of AI technologies.
Score: 0.4435 | Doc: Artificial intelligence and machine learning are transforming industries.
Score: 0.0577 | Doc: A warm cup of coffee in the morning makes me happy.
Score: 0.0435 | Doc: The quick brown fox jumps over the lazy dog.
```

## 2. Pure Sparse Vector Search (Lexical / Keyword)

Sparse vectors have thousands or millions of dimensions, but **most values are zero**. Traditional sparse search (like BM25) counts word frequencies and applies statistical weighting. Modern sparse search (like SPLADE) uses AI to output sparse vectors containing exact words *and* predicted relevant keywords.

- **Pros:** Incredible for exact matches, names, serial numbers, and domain-specific terminology. Extremely fast.
- **Cons:** Completely blind to synonyms unless explicitly trained (e.g., searching "automobile" might completely miss a document saying "car").

### Code Implementation

```python
from rank_bm25 import BM25Okapi
import numpy as np

# 1. Sample Documents
documents = [
    "The quick brown fox jumps over the lazy dog.",
    "Artificial intelligence and machine learning are transforming industries.",
    "A warm cup of coffee in the morning makes me happy.",
    "Deep neural networks are a subset of AI technologies."
]

# 2. Tokenize the corpus (Sparse algorithms look at individual words/tokens)
tokenized_corpus = [doc.lower().split(" ") for doc in documents]
bm25 = BM25Okapi(tokenized_corpus)

# 3. Define and Tokenize the Query
query = "machine learning AI"
tokenized_query = query.lower().split(" ")

# 4. Calculate BM25 Scores
sparse_scores = bm25.get_scores(tokenized_query)

# 5. Rank and Print Results
ranked_indices = np.argsort(sparse_scores)[::-1]

print("\n--- Sparse Search (BM25) Results ---")
for idx in ranked_indices:
    print(f"Score: {sparse_scores[idx]:.4f} | Doc: {documents[idx]}")
```

**Example Output:**

```
--- Sparse Search (BM25) Results ---
Score: 0.9412 | Doc: Artificial intelligence and machine learning are transforming industries.
Score: 0.3956 | Doc: Deep neural networks are a subset of AI technologies.
Score: 0.0000 | Doc: A warm cup of coffee in the morning makes me happy.
Score: 0.0000 | Doc: The quick brown fox jumps over the lazy dog.
```

## 3. Hybrid Search (Dense + Sparse Combined)

Runs **both** a dense search and a sparse search simultaneously, then merges the results. This delivers semantic understanding *and* exact keyword matching.

To merge scores effectively (BM25 scores can be >20 while Dense scores are between 0 and 1), the industry-standard algorithm is **Reciprocal Rank Fusion (RRF)**. RRF scores documents based on their *position* in both result lists rather than raw scores.

### Code Implementation

```python
def rrf(dense_rankings, sparse_rankings, k=60):
    """
    Reciprocal Rank Fusion (RRF)
    k: A constant penalty parameter (standard is 60)
    """
    rrf_scores = {}
    
    def add_ranks(rankings):
        for rank, doc_idx in enumerate(rankings):
            if doc_idx not in rrf_scores:
                rrf_scores[doc_idx] = 0.0
            rrf_scores[doc_idx] += 1.0 / (k + (rank + 1))

    add_ranks(dense_rankings)
    add_ranks(sparse_rankings)
    
    sorted_docs = sorted(rrf_scores.items(), key=lambda item: item[1], reverse=True)
    return sorted_docs

# Ranked indices from previous Dense and Sparse runs:
dense_ranked_indices = [1, 3, 0, 2]   # AI docs ranked highest by Dense
sparse_ranked_indices = [1, 0, 3, 2]  # Exact match for 'learning AI' favored doc 1

# Merge using RRF
hybrid_results = rrf(dense_ranked_indices, sparse_ranked_indices, k=60)

print("\n--- Hybrid Search (RRF) Results ---")
for doc_idx, score in hybrid_results:
    print(f"RRF Score: {score:.5f} | Doc: {documents[doc_idx]}")
```

**Example Output:**

```
--- Hybrid Search (RRF) Results ---
RRF Score: 0.03279 | Doc: Artificial intelligence and machine learning are transforming industries.
Score: 0.03252 | Doc: Deep neural networks are a subset of AI technologies.
Score: 0.03226 | Doc: The quick brown fox jumps over the lazy dog.
Score: 0.03125 | Doc: A warm cup of coffee in the morning makes me happy.
```

## 4. Multi-Vector Search (ColBERT Style)

Instead of squashing an entire sentence into a single dense vector, Multi-Vector search creates **a separate dense vector for every single token (word)** in the document. At query time, it compares every token in the query against every token in the document (called **late interaction**).

- **Pros:** Exceptional accuracy. Retains granular details of specific words while keeping deep contextual meaning.
- **Cons:** Huge storage footprint (vectors multiply exponentially) and slower search speeds.

### Code Implementation Concept

While fully setting up ColBERT requires specialized engines like `PLAID`, the "Late Interaction" concept can be simulated using PyTorch/Sentence-Transformers by extracting token-level embeddings.

```python
import torch
from transformers import AutoTokenizer, AutoModel

# 1. Load a model capable of returning token-level embeddings
tokenizer = AutoTokenizer.from_pretrained("sentence-transformers/all-MiniLM-L6-v2")
model = AutoModel.from_pretrained("sentence-transformers/all-MiniLM-L6-v2")

def get_token_embeddings(text):
    inputs = tokenizer(text, return_tensors="pt", padding=True, truncation=True)
    with torch.no_grad():
        outputs = model(**inputs)
    # Extract the last hidden state: Shape (1, number_of_tokens, hidden_dimension)
    return outputs.last_hidden_state[0]

# 2. Get embeddings for document and query token-by-token
doc_tokens = get_token_embeddings("Artificial intelligence is transforming industries.")
query_tokens = get_token_embeddings("AI impact on industry")

# 3. Late Interaction: MaxSim operator
# Compute similarity matrix between all query tokens and all doc tokens
sim_matrix = torch.matmul(query_tokens, doc_tokens.T) / (
    torch.norm(query_tokens, dim=1, keepdim=True) * torch.norm(doc_tokens, dim=1)
)

# For each query token, find its maximum similarity to ANY document token, then sum
max_sim_per_query_token, _ = torch.max(sim_matrix, dim=1)
multi_vector_score = torch.sum(max_sim_per_query_token).item()

print(f"\nMulti-Vector Late Interaction Match Score: {multi_vector_score:.4f}")
```

**Example Output:**

```
Multi-Vector Late Interaction Match Score: 4.1458
```

## Summary: Search Paradigm Selection Guide

| Search Style | Best Used For | Speed / Resource Cost | Implementation Complexity |
| --- | --- | --- | --- |
| **Pure Dense** | Synonyms, conversational questions, cross-lingual search. | Medium | Easy (`sentence-transformers`) |
| **Pure Sparse** | Exact codes, product SKUs, unique names, logs. | Extremely Fast / Low | Very Easy (`rank-bm25`) |
| **Hybrid** | General production RAG apps requiring balanced accuracy. | Medium | Moderate (Requires RRF merging) |
| **Multi-Vector** | Ultra-high accuracy applications where context nuances matter. | Slow / Very High Storage | Complex (Needs specialized DBs) |

## Hybrid Search in Practice: Pinecone vs. Chroma

Both Pinecone and Chroma can perform hybrid search in a **single call** — no manual client-side combination required. However, they differ in what they expect from the caller.

### Pinecone: Unified Index, Single Call

A specialized index stores both a dense vector and a sparse vector on the same record. Both vectors are passed into **one API request**; Pinecone calculates both scores, applies the `alpha` weighting formula, and returns a single combined result list.

```python
# A single call handles the full hybrid search on the server side
results = index.query(
    vector=dense_query_embedding,          # Generated by your dense model (e.g., BERT)
    sparse_vector=sparse_query_dict,      # Generated by your sparse encoder (e.g., BM25/SPLADE)
    top_k=10
)
```

> **Catch:** Pinecone is purely a *vector* database. It doesn't know what raw text is. The caller is responsible for generating the dense vector array and the sparse token dictionary locally before sending the request.

### Chroma: Automated Hybrid Search, Single Call

Chroma acts as both an embedding manager and a document indexer. Using its `Search` API, a single strategy handles the dual-engine indexing and RRF matching.

```python
from chromadb import Search, Rrf, Knn

# 1. Define how to combine them natively in Chroma
hybrid_rank = Rrf(
    ranks=[
        Knn(query="AI infrastructure optimization"),                       # Dense
        Knn(query="AI infrastructure optimization", key="sparse_field")     # Sparse
    ]
)

# 2. Package into a single search request
search_query = Search().rank(hybrid_rank).limit(10)

# 3. Execute the single hybrid call
results = vector_store.hybrid_search(search_query)
```

> **Advantage:** Chroma automatically manages text tokenization, native full-text indices, and semantic embeddings. Just pass the text string.

**Summary:**
- **Pinecone:** Single DB call, but **you** compute the dense arrays and sparse dictionary before the request.
- **Chroma:** Single DB call, and **Chroma** handles generating the query representations and text indexing internally.

## The Architecture Shift: Split vs. Unified Kernels

### Legacy "Split-Architecture" Anti-Pattern

Historically, vector databases (like early Pinecone) only accepted numerical arrays and had zero concept of text. This forced a complex workflow:

1. Generate dense embedding --> Query **Pinecone** --> Returns Top 50 **IDs** (no text).
2. Send query text --> Query **Elasticsearch/DynamoDB** --> Returns Top 50 Keyword Matches.
3. **Client-Side Stitching:** Custom server code to deduplicate, normalize different scoring systems, and fetch raw text content from DynamoDB/SQL based on IDs.

**Pain Points:** Double network latency, data synchronization bugs across DBs, heavy boilerplate code.

### Modern "Unified" Vector Engine

Modern vector databases (Pinecone, Chroma, Milvus, pgvector) have evolved into multi-modal search engines. They allow defining schemas containing **both numerical vector fields and raw text fields** in the same record.

**Query Lifecycle:** A single text query or vector payload is passed to the database. The DB executes the semantic, keyword, or hybrid lookup on its unified backend and returns a single, sorted response containing the score *and* the actual text string.

## Traditional BM25 vs. AI Sparse (SPLADE)

A key distinction in search engineering: text-based keyword search and "Sparse Vectors" represent two distinct eras of lexical retrieval.

### A. Traditional BM25 (Plain Text Input)

- **How it works:** Text is stored. Queries use plain text (e.g., `"where is AI used?"`).
- **The Tech:** The DB builds an internal *Inverted Index* (mapping exact words to document IDs) and weighs them strictly using statistical math (TF-IDF). Rigid — looks only for exact alphanumeric matches.

### B. Sparse Core / Learned Lexical (Sparse Vector Input)

- **How it works:** The text query passes through an upstream transformer model like **SPLADE** *before* hitting the database. The model outputs a mathematical dictionary of token indices and weights.
- **Term Expansion:** Because it runs through a neural network, the model injects highly relevant synonyms and contextual terms into the sparse vector that weren't in the original text.
  - *Example:* The document text *"We fixed the bug in our server cluster"* automatically expands to include weights for terms like `patch`, `software`, and `kubernetes`.
- **Can Sparse Core replace Dense search entirely?** Almost, but not quite. While Sparse Core handles synonyms beautifully via word injection, **Dense search is still required** for mapping abstract concepts, tracking long-form contextual semantics, and handling cross-lingual data where exact word mappings do not exist.

## Database Implementation Summary

- **Pinecone:** Unified indexing. Supports passing explicit `dense_vector` and `sparse_vector` arrays in a single API call, blended on the server side using a flexible `alpha` slider ($\alpha=1$ is pure dense, $\alpha=0$ is pure sparse). Also supports native Full-Text Search schemas.
- **Chroma:** Developer-friendly all-in-one. Handles text tokenization and embedding generation directly. The native `Search` API handles dual-engine indexing and RRF matching automatically.
- **pgvector + PostgreSQL:** Relational database powerhouse. Leverages native Postgres text engines (`tsvector` for BM25) alongside the `pgvector` extension. Enables hybrid queries and strict RRF blending via standard SQL — no isolated vector infrastructure cluster needed.
